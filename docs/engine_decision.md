## 0. 結論を出すために必要なこと

このゲームは、よくよく見ると **「自動戦闘という名のロジック実行」+「UI 中心の編集・閲覧画面」** で構成される。
"ゲームらしさ"の比重は低めで、見た目は記号的・幾何的でよいと企画書で決めている。
だから判断軸は「ゲームエンジンを使うか、それともアプリとして作るか」が一番大きい。

## 4. 各案の詳細

### 4.A Godot 4
**強み**
- 本格的なゲームエンジン。シーン／ノード／アニメ／物理が標準装備。
- 将来 PC/モバイルネイティブにも書き出せる（v1.1 以降の選択肢として残せる）。
- 公式ドキュメントが整っており、学習素材は豊富。

**弱み**
- **AI コード生成の精度が TS より落ちる**（GDScript は学習データが少ない）。
- Web 書き出し（HTML5）はサイズが大きく（数十 MB）、初回ロードが重い。スマホブラウザ環境で挙動差が出やすい。
- UI（メニュー・編集画面）はノードベースで作れるが、**Web アプリ的な作り込みは React より痛い**。
- このゲームの 8 割を占める「ガンビット編集 UI」を Godot で作るのはオーバーヘッドが重い。

**こういう人向け**
- ゲーム開発を本格的に学びたい／将来別タイトルもエンジンで作りたい。
- ネイティブ展開を v1.1 以降で狙いたい。

---

### 4.B TypeScript + Phaser 3
**強み**
- Web 向けゲーム作りの定番。スプライト・アニメ・タイルマップ・サウンドが揃う。
- TypeScript なので AI コード生成と相性良し。
- バンドルサイズも適正（数 MB）。

**弱み**
- **UI（編集画面・メニュー）が苦手**。Phaser 内で UI を作るより、HTML+CSS で作って重ねる人が多い。
- 結局 React 等を併用することになり、構成が複雑化する可能性。
- このゲームに対してはやや過剰（リッチな描画は要らない）。

**こういう人向け**
- 戦闘画面に**スプライトアニメや派手な演出**を入れたい。
- 「Web ゲームを作っている」という体感が欲しい。

---

### 4.C TypeScript + React（Web アプリ風）
**強み**
- このゲームの中身（**メニュー／編集／ログ表示／ステータス管理**）は Web アプリの得意技そのもの。
- AI（Claude 等）が一番得意な領域。React + TS のコード生成は精度・量ともにトップ。
- バンドルが軽い（1MB 前後）。スマホブラウザで起動が速い。
- **PWA 対応が容易**（Vite + vite-plugin-pwa などで数行）。
- 戦闘画面はテキスト＋CSS アニメ＋簡単な SVG/Canvas で十分に成立する（記号的アートでいくと決めている本作と相性◎）。
- ガンビット評価器を**完全な純粋関数**として TS で書け、ユニットテストが書きやすい。

**弱み**
- 派手な演出は自前で書く必要がある（が、本作には不要）。
- 「ゲーム作ってる感」が薄いので、モチベ的に物足りない可能性。
- 将来ネイティブ書き出しに行きたくなったら、React Native or Tauri 経由などで一手間。

**こういう人向け**
- ゲーム開発の派手さより、**ロジックと UI の作り込み**を楽しみたい。
- 個人開発でとにかく**完成優先**。

---

### 4.D TypeScript + PixiJS
**強み**
- 描画ライブラリとして軽量・高速。
- カスタムレンダリングの自由度が最大。

**弱み**
- 「エンジン」ではないため、状態管理・シーン管理・UI などすべて自前で実装する必要がある。
- 個人開発の労力対効果が悪い。本作で必要な"描画"は React+CSS で足りる。

**こういう人向け**
- 描画にこだわりがあり、エンジン制約を嫌う。
- （本作には基本的に過剰な選択肢）

---

## 5. 決定：C 案（TypeScript + React + Vite + PWA）

> **2026-05-04 確定。**以降の開発はこのスタックを前提とする。

**第1候補（採用）：C. TypeScript + React + Vite + (vite-plugin-pwa)**
理由：
- このゲームの 8 割は UI と状態遷移。React の本来の用途と完全に一致する。
- AI コード生成の精度が一番高い。**個人開発 + AI** という体制を最大限活かせる。
- Web/モバイルブラウザで軽快に動き、PWA 化も簡単。
- ガンビット評価器を**型と単体テストで縛った純粋関数**として実装するのに最適。
- M1 完了基準（コンソール出力で動く戦闘ループ）に最短距離で到達できる。

**第2候補：A. Godot 4**
理由：
- 「将来ネイティブ展開もしたい」「本格ゲーム開発の経験を積みたい」という志向があれば。
- ただし M1〜M3 は React 案より明らかに重い。

**非推奨：B. Phaser / D. PixiJS**
- B：UI が主体の本作には噛み合わない。React 併用で複雑化するくらいなら C 一本のほうが早い。
- D：抽象度が低すぎて、個人開発で自作する量が増える。

## 6. 推奨スタック（C 案を選んだ場合の具体）

```
言語       : TypeScript 5.x
UI         : React 18
ビルド     : Vite 5
ルーティング: react-router 6
状態管理   : 最初は useState/useReducer。複雑化したら Zustand
スタイル   : CSS Modules または Tailwind v3（モバイル UI 作りやすさで Tailwind 推奨）
テスト     : Vitest + Testing Library
PWA        : vite-plugin-pwa
描画/演出  : SVG + CSS アニメ（戦闘の派手さが要れば後から Canvas/Pixi 部分導入）
ホスティング: Cloudflare Pages（無料・PR プレビュー有）
リポジトリ : GitHub（Private 開始 → 公開時 Public）
```

> **C+Pixi の部分導入**：戦闘ログの "演出" だけを後で Canvas/Pixi に置き換える、という拡張パスを残しておけば、見栄えが欲しくなった時に詰まない。

## 7. 決定するための問い（しょうたさんへ）

以下に答えれば自然に決まる：

1. **既に触ったことがあるエンジン／フレームワークはあるか？**
   - Unity / Godot / React / 何もない、など
2. **このプロジェクトを通じて、何を一番得たいか？**
   - (a) ゲームを完成させた実績／(b) ゲーム開発の経験値／(c) AI と共に作る方法論
3. **将来 v1.1 以降でネイティブアプリ化したい願望は強いか？**
   - 強い → Godot が有利／弱い・未定 → React で OK
4. **戦闘画面の派手さの希望は？**
   - "テキスト＋簡単アニメで十分" → React／"スプライトが動いてほしい" → Phaser/Godot
5. **学習コストにどれだけ予算があるか？**
   - "M1 でエンジン学習に1〜2週使ってもいい" → Godot 可／"なるべく既知or薄いほうがいい" → React
