### M3-G が超過した理由

当初の M3-G スコープは「バランス調整 + 統合テスト」だけだったが、実機を触る中で発覚した課題への対応が積み重なった：

1. **G-6**：RulePicker のドロップダウン再発バグ
2. **G-7**：戦闘終了後の UX（次の深度へ遷移）
3. **G-8〜G-11, G-16, G-19**：日英ローカライズ（v1.0 スコープだが M5 想定だった）
4. **G-12**：Portal 化（Chromium の select クリップ対策）
5. **G-13〜G-14**：ジョブ変更 UI（M2-E で見落としていた）
6. **G-15, G-18**：select → button 群への全置換（dropdown 問題の根本解決）
7. **G-17**：プリセット強化（HOLY_BOLT のデフォルト化）

**学び**：「手動デモで気づく問題」は計画段階では予測しきれない。実機検証の時間を **Phase 名目の所要に明示的に組み込む** べきだった。

## 3. うまくいったこと

### 3.1 テスト先行による安心感

新機能追加のたびに 100 行単位のテストを書いた結果、**272 tests 全 PASS** を維持できた。リファクタ（select → button、ローカライズ等）も恐れずできた。

### 3.2 balance.test.ts による「数字で示す」設計

「DEMON_LORD は標準パーティで 19 ターン勝利」「装備なしでは depth 10 を負ける」のような数字を残したことで、調整の方向性が明確に。

### 3.3 i18n 辞書を flat に作った

`STRINGS["common.save"] = { ja, en }` というシンプルな構造で、追加・参照が直感的。react-i18next を入れなくて正解だった。

### 3.4 Portal 化と button 群への統一

中盤の悩みどころだった select クリップ問題を、「**このアプリでは native select を使わない**」というルールに統一して根本解決。今後ドロップダウン関連のバグ報告は起きないはず。

## 4. 苦戦したこと

### 4.1 Chromium の native select clip 問題（最大の沼）

`<main overflow-auto>` の中に置いた `<select>` が、popup を clip されて開かなくなる。**5 回くらい違う形で報告された**。

- M3-D：EquipmentPicker（`max-h-[90vh]` + `overflow-auto` が原因）
- M3-G-6：RulePicker で同じパターンを再発
- M3-G-14：PartyScreen の job select
- M3-G-15：EquipmentPicker（Portal 化しても popup 不安定）
- M3-G-18：RulePicker の EnumSelector

最終的に「**select 全廃 + toggle button 群に統一**」で根絶。最初からこの方針で行けば 2 日分くらい節約できた。

### 4.2 i18n 化で既存テストが連鎖的に壊れる

UI 文字列を t() に置き換えるたびに、テストの期待値（`/INTERPOSE/`、`/SELF_HP_LT/` 等）が壊れた。1 回の修正で十数件のテストを更新する場面が複数回。

**学び**：i18n 導入は早ければ早いほど良い。**M2 のうちに最低限の t() 基盤だけでも入れておけば**、M3 でこの手戻りはなかった。

### 4.3 ジョブ変更 UI の見落とし

PartyScreen が「装備変更しかできない」状態だったことに、depth 10 が突破できないユーザー体験で初めて気づいた。M2-E のスコープに入っていたはずなのに、剣士単一で済ませてしまっていた。

**学び**：v1.0 で 3 ジョブと言いつつ「ジョブ選択 UI」がなかったのは画竜点睛を欠いた。次フェーズの計画書は「UI と機能の両方が揃っているか」を Phase 完了基準に明記する。

## 5. M4 への申し送り

### 5.1 そのまま使える資産

- balance.test.ts のシミュレーションは MVP 公開後のバランス調整にもそのまま使える
- i18n 辞書は新規ストリングを足すだけで済む構造
- presetBeginner / presetTank / presetFinisher / presetExploitWeakness は「初心者がそのまま使える」状態に磨かれた

### 5.2 M4 で対応したい技術的負債

- `Layout.tsx` の `<main overflow-auto>` は select 問題のトリガになっていたが、button 群に統一したので **今は無害**。M4 で PWA 化するときに `position: sticky` に切り替えるのが望ましい
- 戦闘ログの formatEvent は string ベースで組み立てている。M4 でリプレイ機能を作るなら React コンポーネント化する方が表現の幅が広がる
- `presetBeginner` のルール数が 3 → 4 に増えた。プリセット名「初心者向け」と実態（ボス対応込み）がやや乖離。M4 でリネーム検討
